=== VIBEMATCH: CORE SYSTEM DESIGN ===
(Conceptual Architecture - Weights Dynamically Adjustable)

---

## 1. CORE CONCEPT

VIBEMATCH is a multi-dimensional semantic matching system that:
- Represents users as vectors across 4 independent layers
- Assigns users to dynamic, overlapping sets (not single buckets)
- Uses hierarchical clustering for scalable matching
- Applies game-theoretic algorithms to guarantee stable pairs
- Learns and improves through ensemble feedback loops

Key insight: Matching is a multi-dimensional optimization problem, not a simple scoring system.

---

## 2. FOUR LAYERS OF USER DATA

### Layer 1: Music/Taste
Vector representation of user preferences (384-dimensional semantic embeddings).
Each user becomes a point in vector space. Similarity = compatibility.

### Layer 2: Identity/Demographics
Structural data: sex, gender, sexuality, age, location.
Used for coarse filtering and set assignment.

### Layer 3: Behavioral/Personality
Embeddings from user interactions: bio, prompts, swipes, conversations.
Captures implicit preferences and attraction patterns.

### Layer 4: Preference/Goals
What user is explicitly looking for. Can be text, vectors, or tags.

All 4 layers are independent. They can be activated or combined in different ways.

---

## 3. MULTI-SET CLUSTERING SYSTEM

Traditional approach: User in 1 set -> limited flexibility  
VIBEMATCH approach: User in 4+ sets simultaneously -> nuanced matching

### How It Works

**Set Types**
- Demographic Sets (age range, location, gender)
- Taste Sets (music similarity clusters)
- Behavioral Sets (engagement level, profile richness)
- Preference Sets (looking for what type of person)

**User Assignment Example**
```text
User A belongs to:
|- Set-D1 (demographic match)
|- Set-M1 (music taste match)
|- Set-B1 (behavioral fit)
\- Set-P1 (preference alignment)
```

**Why Multi-Set?**
- Flexible: User is not trapped in one box
- Nuanced: Different sets capture different compatibility dimensions
- Dynamic: Can add or remove sets as the user evolves
- Explainable: Can show "you matched because of music" vs "because of location"

---

## 4. HIERARCHICAL CLUSTERING

Sets are grouped into clusters (higher abstraction level).

```text
Cluster 1: "Young Active Indie Lovers"
|- Set-D1 (young, city-based)
|- Set-M1 (indie music)
\- Set-B1 (active engagement)

Cluster 2: "Selective Hip-Hop Community"
|- Set-D2 (different age/location)
|- Set-M2 (hip-hop preference)
\- Set-B2 (selective engagement)
```

**Benefits**
- Scalability: O(log n) instead of O(n)
- Coherence: Can measure internal set consistency
- Quality gates: Detect and remove bad clusters automatically

---

## 5. MATCHING PIPELINE (7 STAGES)

### Stage 1: Identify User's Sets
Get all sets the user belongs to.
Check set quality (sufficient data? good match rate?).

### Stage 2: Create Candidate Pool
Union of all users in the user's sets.
Apply spatial/semantic clustering to reduce.

### Stage 3: Coarse Filtering
Use Delaunay triangulation (spatial clustering on demographics).
Use vector clustering (DBSCAN on semantic embeddings).
Reduce from 2000 -> 300 candidates.

### Stage 4: Scoring
Calculate compatibility scores using all available layers.
Could be: music similarity, behavioral alignment, preference match, etc.
Scoring formula is flexible, not fixed.

### Stage 5: Stable Matching
Apply Gale-Shapley algorithm on top candidates.
Both user and candidates rank each other.
Algorithm produces stable matches (mutual preference).

### Stage 6: Ensemble Ranking
Combine multiple ranking approaches:
- Pure score ranking
- Behavioral clustering insights
- ML prediction (if available)
- Cluster compatibility

Return top N candidates.

### Stage 7: Return to User
Show 20-50 ranked candidates.
User swipes/likes -> feedback loop improves system.

---

## 6. DYNAMIC SCORING SYSTEM

Scoring is flexible and adjustable, not hardcoded.

Could combine:
- Music similarity (cosine of vectors)
- Behavioral fit (interaction patterns)
- Preference alignment (explicit match)
- Location compatibility
- Personality match
- etc.

**How it's calculated**
```text
composite_score = FUNCTION(all_layers, user_context)
```

Weights can be:
- Dynamic (change per user, per phase)
- Learned (ML model finds optimal weights)
- Explicit (admin sets for A/B testing)
- Hybrid (base weights + user overrides)

**Advantage**
Fully adaptable. As you learn what works, adjust scoring.

---

## 7. TIME-STAMPED ONBOARDING

Progressive data collection, tracked precisely:

```text
T=0:      User signs up (Onboarding Set 0)
T=5min:   Music data added (Layer 1 complete, Onboarding Set 1)
T=15min:  Demographics filled (Layer 2 complete, Onboarding Set 2)
T=25min:  Bio/prompts done (Layer 3 complete, Onboarding Set 3)
T=25min:  Ready for matching (transition to final sets)
```

**Benefits**
- Can show partially-ready users to others (with sigmoid visibility ramp)
- Track data quality over time
- Identify drop-off points in onboarding
- Timestamp data for freshness metrics

---

## 8. ENSEMBLE LEARNING & FEEDBACK

After a user has interaction history (20+ swipes), activate ensemble:

**Component 1: Historical Ranking**
Keep top matches from previous rounds.
See if the user interacts with them.

**Component 2: Behavioral Learning**
Analyze: "User A tends to swipe right on X type"
Adjust future candidate rankings accordingly.

**Component 3: ML Model**
Train on: (user features, candidate features, outcome) -> prediction
Predict which matches will lead to conversations.

**Ensemble combines all 3** -> better predictions over time.

**Feedback Loop**
More interactions -> Better data -> Better ML -> Better matches -> More interactions.

---

## 9. USER EVOLUTION & MOBILITY

Users do not stay static. Sets reclassify:

```text
Day 1: User A in Set-B2 (selective, <5 matches)
Day 30: User A made 8 matches, 3 conversations
Action: Reclassify -> Set-B1 (active, >10 matches possible)

Next matching run: Different candidate pool (more active users shown)
```

**System detects**
- Changes in behavior (engagement level shift)
- Data enrichment (added photos, more profile info)
- Interaction patterns (preference trends)

**User controls**
- Can opt into new sets ("Show me more creative types")
- Can opt out of sets ("Don't show me X")

---

## 10. QUALITY GATES

Automatic detection of bad sets/clusters:

```text
IF set has <10 users:
    -> BAD_SET (too small)

IF set has <0.3 average profile richness:
    -> BAD_SET (insufficient data)

IF set has <0.1 match success rate:
    -> BAD_SET (not working)

IF set has no activity in 90 days:
    -> STALE_SET (needs refresh)
```

When a bad set is detected:
1. Mark as inactive
2. Do not use for matching
3. Option A: Merge with similar healthy set
4. Option B: Rebuild with new criteria
5. Re-validate cluster integrity

## 11. SCALABILITY & ARCHITECTURE

### Complexity Analysis
```text
Onboarding:         O(n) per user
Set lookup:         O(1) hash
Clustering:         O(n log n) Delaunay + DBSCAN
Scoring:            O(n) per user
Gale-Shapley:       O(n^2) worst, O(n log n) typical (optimizable)
Ensemble:           O(n) prediction
Overall:            O(n log n) with proper indexing
```

### Database
- Current: SQLite (development)
- Target: PostgreSQL with pgvector extension
- Indexes on: user_id, set_id, location, vectors

### Scalable to
- 100K users (Phase 1)
- 500K users (Phase 2)
- 2M users (Phase 3)
- 10M+ users (Phase 4, with clustering efficiency)

## 13. KEY DESIGN PRINCIPLES

1. **Vector-First**: Every user dimension represented as a vector
2. **Layered**: Independent data layers can be combined flexibly
3. **Multi-Set**: Not single-bucket assignment (more nuanced)
4. **Stable**: Game-theoretic guarantee of no unstable pairs
5. **Dynamic**: Weights, sets, clusters evolve over time
6. **Quality-Gated**: Automatic detection and removal of bad data
7. **Learning**: Ensemble approach improves with data
8. **Transparent**: Can explain "you matched because of X, Y, Z"
